# 01_V3整体变化说明

V2 已经是前端 + C++ 后端项目。

V3 的目标是在不改变技术栈难度的前提下，加入用户注册和登录。

## 1. 为什么要加登录注册

如果没有登录注册，所有任务都属于同一个系统。

这会带来问题：

- 不同同学无法区分自己的任务。
- 所有人看到同一份任务列表。
- 统计数据也混在一起。

加入登录注册后，每个用户都有自己的任务数据。

## 2. V3 新增了哪些文件数据

### users_v3.txt

保存用户信息。

格式：

```text
username|password|displayName|createdDate
```

### study_tasks_v3.txt

保存任务信息。

V3 比 V2 多了 username 字段。

格式：

```text
id|username|title|topic|dueDate|dueTime|priority|needReview|completed|createdDate|completedDate|estimatedMinutes|note
```

## 3. V3 新增了哪些接口

| 接口 | 作用 |
|---|---|
| POST /api/register | 注册用户 |
| POST /api/login | 登录用户 |

其他接口仍然存在，但多数都增加了 username 参数。

例如：

```text
GET /api/tasks?username=alice
GET /api/stats?username=alice
GET /api/reminders?username=alice
```

## 4. V3 的前后端分工

### 前端负责

1. 显示登录注册页面。
2. 收集用户名和密码。
3. 调用注册接口。
4. 调用登录接口。
5. 登录成功后保存当前用户。
6. 所有任务请求都带上 username。
7. 退出登录时清除当前用户。

### 后端负责

1. 保存用户。
2. 校验用户名和密码。
3. 注册时判断用户名是否重复。
4. 登录时判断密码是否正确。
5. 添加任务时绑定 username。
6. 查询、统计、提醒时按 username 过滤任务。

## 5. V3 为什么仍然不使用数据库

因为你的课程基础是 C++ 基础语法和数据结构。

如果突然加入 MySQL 或 SQLite，项目难度会变高。

所以 V3 仍然使用文本文件。

这既能体现数据持久化，又不会超出太多学习范围。
